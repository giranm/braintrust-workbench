## Documentation Workflow

### For Coding Agents

When working on this project:

1. **Before starting**:
   - Read `planning.md` for project context
   - Check `issues.md` for known problems
   - Review `implementation.md` for technical decisions

2. **During development**:
   - Update `implementation.md` with decisions and progress
   - Log new issues in `issues.md`
   - Reference docs when making technical choices

3. **After completing work**:
   - Update `changelog.md` with changes
   - Move resolved issues in `issues.md`
   - Update planning status if applicable

### For Developers

This documentation structure ensures:
- ✅ Knowledge is captured in version control
- ✅ Context is available for AI-assisted development
- ✅ Decisions are documented with rationale
- ✅ Issues and progress are tracked
- ✅ Project history is preserved

## Tips

- **Keep docs updated**: Stale docs are worse than no docs
- **Be concise**: Focus on "why" not just "what"
- **Link to code**: Reference specific files and line numbers
- **Date entries**: Always include dates for context
- **Use checklists**: Track progress with `- [ ]` items
