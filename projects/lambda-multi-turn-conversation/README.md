# Braintrust Multi-Turn Lambda Tracing

This project is a **reference implementation** that demonstrates multi-turn conversation tracing for AWS Lambda architectures using Braintrust. It shows how to create a single hierarchical trace across multiple stateless Lambda invocations, where each message in a conversation becomes a child span under a shared root trace.

> **Note**: This is a demonstration project intended to be adapted for your specific requirements. Terraform uses local state for simplicity - production deployments should use a remote backend (e.g., S3). See [Adapting for Production](#adapting-for-production) for guidance.

## Architecture

```
User Input → Python CLI Client → AWS Lambda (Node.js) → DynamoDB → Braintrust
                                        ↓
                                  OpenAI gpt-4o-mini
```

### Components

1. **Lambda Function** (Node.js/TypeScript)
   - Handles conversation messages
   - Creates traced spans with Braintrust SDK
   - Manages parent/child span relationships via `span.export()`
   - Stores conversation state in DynamoDB

2. **DynamoDB Table**
   - Stores conversation state between invocations
   - Keys: `conversation_id`
   - Attributes: `root_span_export`, `last_span_export`, `message_count`

3. **Python CLI Client**
   - Interactive console for testing conversations
   - Invokes Lambda via AWS SDK (boto3)
   - Manages conversation history

4. **Terraform Infrastructure**
   - Provisions all AWS resources
   - Consistent tagging across resources
   - Local state management

5. **Mise**
   - Manages runtime versions (Node.js 20, Python 3.12, Terraform 1.7)

## How It Works

### Multi-Turn Tracing Flow

1. **First Message (Message #1)**
   ```typescript
   // Creates root "conversation" span with turn-1 nested inside
   traced(async (conversationSpan) => {
     conversationSpan.log({ input, metadata });

     result = await traced(async (turnSpan) => {
       const response = await callLLM(messages);
       turnSpan.log({ output, metrics });
       return response;
     }, { name: "turn-1", type: "llm" });

     const rootExport = await conversationSpan.export();
     await saveState({ root_span_export: rootExport, ... });
   }, { name: "conversation", type: "task" });
   ```

2. **Subsequent Messages (Message #2, #3, ...)**
   ```typescript
   // Creates turn span as child of root conversation
   const state = await getState(conversationId);

   traced(async (span) => {
     span.log({ input, metadata });
     const response = await callLLM(messages);
     span.log({ output, metrics });
     return response;
   }, {
     name: "turn-2",
     type: "llm",
     parent: state.root_span_export  // Links to root conversation span
   });
   ```

### Result in Braintrust UI

A 3-message conversation appears as:

```
Root Trace: "conversation" (type: task)
├─ Child: "turn-1" (type: llm)
│  └─ OpenAI API call (auto-traced by wrapOpenAI)
├─ Child: "turn-2" (type: llm, parent: conversation)
│  └─ OpenAI API call
└─ Child: "turn-3" (type: llm, parent: conversation)
   └─ OpenAI API call
```

All under a single `root_span_id`, creating a cohesive conversation view.

## Configuration

### Environment Variables

Copy `.env.example` to `.env` and fill in your values. Each variable is set once - the Makefile automatically derives `TF_VAR_` exports for Terraform:

```bash
# AWS
AWS_PROFILE=sandbox
AWS_REGION=us-east-1

# Braintrust (REQUIRED)
BRAINTRUST_API_KEY=your_api_key_here
BRAINTRUST_PROJECT_ID=your_project_id_here

# OpenAI (REQUIRED)
OPENAI_API_KEY=your_openai_api_key_here

# Terraform configuration
ENVIRONMENT=dev
PROJECT_NAME=braintrust-multi-turn-conversation
FUNCTION_NAME=braintrust-multi-turn-conversation-lambda
OWNER=your_name
```

### Terraform Variables

Customize infrastructure in `terraform/variables.tf`:

- `ENVIRONMENT` - Environment name (dev/staging/prod)
- `AWS_REGION` - AWS region
- `PROJECT_NAME` - Project name for resource naming
- `FUNCTION_NAME` - Lambda function name
- `LAMBDA_MEMORY_SIZE` - Memory in MB (default: 512)
- `LAMBDA_TIMEOUT` - Timeout in seconds (default: 30)

### Resource Tags

All resources are tagged with:
- `Environment` - From variable
- `Project` - braintrust-multi-turn-conversation
- `ManagedBy` - terraform
- `Owner` - From variable
- `Purpose` - multi-turn-tracing-lambda-example
- `Repository` - braintrust-workbench

## Troubleshooting

### Lambda Deployment Fails

**Error**: `ResourceNotFoundException: Function not found`

**Solution**: Deploy infrastructure first:
```bash
make deploy-infra
```

### Terraform Apply Fails

**Error**: `Error creating DynamoDB Table: ResourceInUseException`

**Solution**: Table already exists. Either:
1. Import existing table: `terraform import aws_dynamodb_table.conversations dev-braintrust-multi-turn-conversation-conversations`
2. Or destroy and recreate: `make destroy && make deploy-infra`

### Client Cannot Invoke Lambda

**Error**: `AccessDeniedException`

**Solution**: Check AWS credentials:
```bash
aws sts get-caller-identity --profile sandbox

# Verify you have permissions for:
# - lambda:InvokeFunction
# - dynamodb:GetItem, PutItem
```

### No Traces in Braintrust

**Possible causes**:
1. **Invalid API key**: Check `BRAINTRUST_API_KEY` in `.env`
2. **Wrong project**: Verify `BRAINTRUST_PROJECT_ID`
3. **Logs not flushed**: Check Lambda logs in CloudWatch

**Check CloudWatch Logs**:
```bash
make logs
```

### Traces Not Linked

**Symptom**: Each message shows as separate trace (not nested)

**Diagnosis**:
1. Check DynamoDB state has `root_span_export`
2. Verify `parent` parameter is passed to `traced()`
3. Check Lambda logs for "hasRootParent: true"

## Adapting for Production

This is a reference implementation designed for demonstration and local development. When adapting for your own use case, consider the following changes:

### Terraform State

This project uses **local state** (`terraform.tfstate` on your machine) for simplicity. For production or team use:

```hcl
# In terraform/main.tf, uncomment and configure the S3 backend:
backend "s3" {
  bucket = "your-terraform-state-bucket"
  key    = "multi-turn-conversation/terraform.tfstate"
  region = "us-east-1"
}
```

### Other Production Considerations

- **API Gateway**: Add an API Gateway or ALB in front of the Lambda for HTTP access
- **Authentication**: Add request signing, API keys, or IAM-based auth
- **Secrets Management**: Use AWS Secrets Manager or Parameter Store instead of Lambda environment variables
- **Monitoring**: Add CloudWatch alarms for error rates and latency
- **DynamoDB**: Add conditional writes for concurrent conversation safety
- **CI/CD**: Replace the Makefile with a pipeline (GitHub Actions, CodePipeline, etc.)
